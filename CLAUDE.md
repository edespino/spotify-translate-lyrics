# CLAUDE.md

Context for working on this repo.

## What this is

Personal web app: shows time-synced Spanish lyrics for the currently playing Spotify track with a line-aligned English translation, Spotify-style scrolling. Two processes started by one `npm run dev` (concurrently): a Vite + React + TypeScript frontend and a small Express + TypeScript server run with tsx.
The dev frontend canonicalizes `localhost` to `127.0.0.1` and uses Vite `strictPort` so Spotify OAuth and localStorage stay on the configured origin instead of drifting to an empty token store.

## Layout

- `src/spotify.ts` Spotify OAuth (Authorization Code with PKCE, no client secret, tokens in localStorage, silent refresh). Polls /me/player/currently-playing every 3s, backs off to 10s on 429. seekTo PUTs /me/player/seek with the same 401-refresh handling; 403/404 (no active device or restricted context) throw SeekUnavailableError, which App shows as a brief notice. Seek requires the user-modify-playback-state scope; sessions connected before it was added must reconnect once.
- `src/sync.ts` PositionTracker interpolates playback position between polls with performance.now(); a poll that disagrees with interpolation by more than 1.5s (seek) or a play/pause flip re-anchors. nudge(positionMs, nowMs) re-anchors immediately after a locally initiated seek (line replay) so the highlight snaps without waiting for a poll, and opens a 2.5s settle window: Spotify applies seeks asynchronously, so during the window a poll anchors only if it agrees with the post-nudge position (confirming the seek); stale pre-seek polls are ignored for anchoring (play/pause flips still apply, anchored at the interpolated position). The first confirming poll or expiry ends the window, so an external seek wins again. App also triggers an immediate poll to reconcile. findActiveLine is a binary search over line timestamps.
- `src/lrc.ts` LRC parser: handles minutes over 9, 1 to 3 digit fractions, multiple timestamps per line, skips malformed lines, keeps empty text lines.
- `src/lyrics.ts` LRCLIB client (https://lrclib.net/api/get, CORS is open, called from the browser). Result kinds: synced, plain, instrumental, none.
- `src/detect.ts` client-side English detection on the full joined lyric text, before any /api/translate call. Pure stopword and character frequency heuristic (English vs Spanish stopword hit rates plus accented characters and inverted punctuation), no network, no library. Biased: only confidently English text skips translation; uncertain or mixed (Spanglish) text translates as before. English songs render read-only in a single centered pane (src/translation.ts holds the fetch-if-needed flow that returns null without any network call in that case).
- `src/App.tsx` state machine: auth, poll loop, per-track lyrics + translation loading, requestAnimationFrame loop for the active line.
- `src/components/LyricsView.tsx` both languages render in ONE scroll container as two-column grid rows, so lines stay vertically aligned by construction; the active row is smooth-scrolled to a 32% anchor with a top clamp (long jumps and reduced-motion users get an instant snap). The Spanish column is right-aligned in dual-pane mode so pairs meet at the central gutter. Rows before the active line get a `past` class whose color follows the past-line mode in appearance settings (dim by default, bright or neutral selectable). Enlarge and the active pop scale CELLS, not the row, so column alignment survives. Rows are focusable (Enter toggles enlarge, F2 or e edits the Spanish cell), carry list/listitem roles, aria-current on the active row, and per-language lang attributes. Double-click edits a cell, click enlarges a row. While translating, EN cells show shimmering skeleton bars (widths cycle 45/60/75% by row; shimmer only under no-preference motion); on translation error they show a dim placeholder instead, and the header shows a prominent retry button. Per-line replay: synced rows with text (not the empty ♪ placeholders, never plain lyrics) get an absolutely positioned replay button in the row's left padding, visible on the active row and on hover/focus, with its own click target (stopPropagation) so row click/double-click/Enter behavior is untouched; `r` on a focused row replays that row (stopPropagation so the global handler stays quiet), `r` globally replays the active line, and both are guarded against text inputs. English single-pane mode keeps the replay affordance. Pure per-row logic (phase classification, EN cell state, key mapping including replay, replay eligibility, typing-target guard, scroll behavior) lives in `src/components/lyricsRow.ts` and is unit tested. Active-recall mode (language learning): a Recall pill in the English pane header (dual-pane only, persisted in localStorage; storage access is best-effort and never throws) blurs EN cells so the listener tries the Spanish first; skeleton, error, and empty placeholder cells are never masked. A masked cell is a focusable button (aria-label announces the hidden translation, blurred text is aria-hidden and unselectable) whose click/Enter/Space reveals only that line (stopPropagation so the row does not enlarge); `t` reveals the active line with the same typing/modifier guards as `r`. After a pointer reveal the button stays mounted unblurred (tail phase) so the rest of that same gesture, identified by event.detail (never wall-clock time), is swallowed: the second click and dblclick of a revealing double-click (detail >= 2) do nothing, while a fresh click (detail 1), mouseleave, or blur releases the mask so clicking enlarges and double-click edits as usual. Reveals reset on track change or when the mode is toggled off and on. Pure logic (mask eligibility, per-track reveal set, `t` mapping, fault-tolerant localStorage round trip, the maskGesture state machine) lives in `src/components/recall.ts` and is unit tested.
- Word gloss (language learning): in the dual-pane view every word in a Spanish cell is a gloss target: hover ~450ms or alt-click shows a small dark popover (word, gloss, partOfSpeech tag, note when non-empty) fetched from POST /api/gloss with the full line as context. Gloss is pointer-first BY DESIGN: word targets are plain inline spans kept out of the tab order (rows already carry the focus semantics; dozens of rows times many words would wreck tabbing), the popover has role tooltip and selectable text. Lines are split with Intl.Segmenter (word granularity, es locale; unicode-aware regex fallback) into tokens that concatenate back to the exact original string, so rendering is pixel-identical to plain text; the only affordance is a subtle underline on hover (transition gated to no-preference motion). Plain click and double-click on a word bubble to the row's enlarge/edit as before; only alt-click is claimed. No targets in EN cells, edit inputs, empty/instrumental lines, or English single-pane mode. The popover is position: absolute inside the scroll container (scrolls with the lyrics), placed below the word, flipped above near the visible bottom, clamped horizontally; one at a time; dismissed by Escape, scrolling the container, mouseleave of word+popover (short grace timer so the pointer can travel between them), track change, starting an edit, or another gloss opening. While loading it shows a small skeleton bar; a 502/network failure shows "no gloss available" briefly; a 400 dismisses silently. `src/gloss.ts` is the client: in-memory cache keyed by normalized word+context (the server also caches on disk; both layers are correct), an in-flight map so hover jitter cannot double-fire, and per-caller AbortController wiring so a dismissed popover's pending fetch is cancelled and its late response never renders. Pure logic (segmentation, cache key normalization, the popover state machine, gesture eligibility, popover placement) lives in `src/components/gloss.ts`; both are unit tested. Glossing works with recall masking on: the popover reveals only the one word's gloss, never the masked EN line.
- Appearance settings: a Settings pill in the NowPlaying header opens `src/components/SettingsPanel.tsx`, a right-side slide-over with the same non-modal interaction pattern as VocabPanel (Escape or outside click closes; opening one panel closes the other since they share the right edge). Two controls: the active-line accent (curated set: green default, amber, sky blue, magenta, white to restore the original look) and the past-line treatment (dim default: past darker than upcoming; bright: the original past-brighter look; neutral: past matches upcoming). Colors reach CSS as one custom property (--accent-active, set inline by LyricsView from the chosen accent) plus one past-* class on .lyrics-view that only sets --past-color, so the row selector structure is unforked; both apply to the English single-pane view too. Color only, never weight, so glyph widths never jitter; the default green is a lighter shade (#4ade80) than the enlarge/focus green (#1db954) so the active line stays distinguishable from the enlarge treatment and from the #d6d6d6 hover in every accent choice, including under reduced motion where the scale cues are absent. Pure logic (settings shape, defaults, per-field validation of persisted values against the allowed sets, fault-tolerant localStorage round trip, the CSS hook mapping) lives in `src/components/settings.ts` and is unit tested.
- `src/components/EmptyState.tsx` shared screen for the non-lyric states (lyrics loading, none, instrumental, lyrics-service error, nothing playing): ~200px album art (dim music-note block when no artwork URL), track title, status line, centered in the lyrics area. The lyrics-service error adds a Retry pill wired to App's loadLyrics, which re-runs the LRCLIB fetch for the current track without waiting for a track change. State-to-message mapping, retryability, and skeleton width cycling are pure functions in `src/components/emptyState.ts`, unit tested.
- Sync-status badge: NowPlaying shows a muted non-interactive "Synced" or "Unsynced" pill (the Unsynced title explains that LRCLIB has no timestamps so there is no moving highlight) only once the lyrics result is known; instrumental, none, loading, and error states show no badge since EmptyState already covers them; the pure state-to-badge mapping lives in `src/components/syncStatus.ts` and is unit tested.
- Mark wrong lyrics: a quiet Mark wrong pill in the NowPlaying header (visible only while synced or plain lyrics show) suppresses a track whose LRCLIB entry is wrong; the server records it in data/marked.json (MarkStore in server/cache.ts, same serialized mutation queue and tmp-then-rename writes) and deletes the cached translation at mark time (delete-on-mark: reset just re-enters the normal fetch-and-translate flow, so a corrected upstream entry gets exactly one fresh translation). While marked the app makes no LRCLIB or translation calls for the track: the marks list loads once per session like vocab, and NO track loads lyrics until that list has resolved (shouldLoadLyrics gates the effect), so a marked track playing at boot never hits LRCLIB even when Spotify polling wins the race; if the marks fetch fails the app proceeds as unmarked (availability over suppression). /api/translate answers 409 for a marked track as a server-side guard, and the client treats that 409 as marked (TrackMarkedError suppresses the lyrics with a fallback record and refreshes the marks list) rather than a translation error. The translate path's cache write re-checks the mark inside the same per-track mutation queue turn (writeUnless), so a mark made while the provider was running cannot have its cache delete undone by the finishing translation. The lyrics area shows the EmptyState with "Lyrics marked as incorrect", a Reset pill, and a Report to LRCLIB button: the server requests a challenge from lrclib.net, solves the SHA-256 proof of work within a bounded budget (server/lrclibFlag.ts, yields to the event loop while searching), and POSTs a community flag with the LRCLIB numeric id captured from the lyrics fetch (persisted on the mark record and, going forward, the cache entry; a mark without an id disables the button with a title explaining that a reset and replay captures it). Endpoints: GET /api/marks, POST /api/mark, POST /api/mark/reset, POST /api/mark/report. Pure logic in src/components/markedLyrics.ts and the solver helpers in server/lrclibFlag.ts are unit tested; the integration tests mock the LRCLIB endpoints and never call the real service.
- `server/app.ts` Express app factory (takes a provider and data dir, which is what the integration tests use).
- `server/translator.ts` provider interface plus the Gemini implementation (@google/genai, model from GEMINI_MODEL, default gemini-flash-lite-latest, JSON array in and out). translateLines enforces the N-in N-out contract: retry the batch once, then fall back to chunks of 20 lines, each strictly validated. A 429 waits for the provider's suggested retry delay (bounded attempts). Any chunk failure fails the whole translation with TranslationFailedError; the server answers 502 and caches nothing. Original text is never returned as a translation.
- `server/cache.ts` disk cache at data/translations/<trackId>.json. Overrides (editedEs/editedEn) always win; retranslate never overwrites en on a line with editedEn.
- Title translation: translateLinesWithTitle prepends the song title as the first line of the same Gemini batch (no extra request; the N-in N-out count includes it; in the chunked fallback the title rides in the first chunk). The result is stored as titleEn on the cache entry and returned by /api/translate. Cache files written before titleEn existed lazily backfill only titleEn on the next cached response using one title-only provider request; failure is ignored so cached lyrics still return, and the next play retries. Retranslate refreshes titleEn. NowPlaying shows titleEn inline after the original title as muted parenthesized text, suppressed when it matches the original case-insensitively (src/translation.ts translatedTitle) and never in English single-pane mode. The title is not editable through the override system.
- API: POST /api/translate, POST /api/retranslate, POST /api/override, POST /api/override/reset, POST /api/gloss, GET /api/translations/:trackId, GET /api/marks, POST /api/mark, POST /api/mark/reset, POST /api/mark/report. Frontend reaches it through the Vite proxy at /api.

## Rules

- `data/` is user data derived from copyrighted lyrics. Never commit it, never weaken the .gitignore entry.
- GEMINI_API_KEY stays server-side only. Never expose it to the browser or prefix it with VITE_.
- Timestamps are never editable, only the text of a line.
- No em dashes or en dashes anywhere in this repo, including docs and commit messages. ASCII punctuation only.
- Commit messages are terse and conventional, no AI attribution of any kind.

## Checks

`npm test`, `npm run typecheck`, `npm run lint` must all pass. Tests are Vitest; server integration tests use supertest with a mocked provider and a temp data dir.
